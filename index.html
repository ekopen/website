Live Trading Engine

Overview

This project develops a full-stack trading engine that integrates real-time data ingestion, processing, and machine learning to drive automated trading decisions.

The system is built around four core modules:

  • Market Data Ingestion: Ingests real-time, tick-level data from a crypto exchange.
  • Market Data Storage: Continuously writes incoming data to a local and cloud-backed database.
  • Machine Learning Service: Retrains machine learning models regularly to adapt to current market conditions.
  • Execution Engine: Executes trades using ML predictions with integrated risk and portfolio management.

Here are some of the key results:

  • High-Throughput Streaming: Processes 100k+ ticks per minute across five cryptocurrencies with sub-second latency.
  • Efficient Storage: Maintains a rolling 30 day window of high-frequency data (over 500 million rows) while archiving older data to the cloud.
  • Adaptive Trading: Retrains five ML models in real time to generate trades across ten portfolio strategies.

Additionally, I took away several lessons from building this:

  • Uptime reveals hidden issues: Persistent logs and crashed connections taught me the importance of graceful restarts and resource monitoring.
  • Architecture matters: Early ad hoc design caused hours of rework, such as migrating to Docker and consolidating modules sprawled across several servers.
  • Modularity drives scalability: Adding new assets or strategies was painful until I fully modularized each component.

Architecture

DigitalOcean hosts the server the project runs on. It currently supported by a $48/month droplet with 8GB RAM, 4 vCPUs, and 160GB SSD storage.

Docker is the platform this project is orchestrated with. Besides the module containers, there are containers for Kafka, Clickhouse, Grafana, Prometheus, and other monitoring tools.

Grafana powers all dashboards for real time monitoring.

Binance serves as the source for cryptocurrency tick-level data, which is ingested from the exchange in real time via a WebSocket provided through Finnhub.

Kafka captures all data from the WebSocket, where a producer publishes incoming messages to a Kafka topic managed by one broker.

ClickHouse is connected to the data stream via a Kafka consumer. New data is appended to the database via batch processing, and monitoring/diagnostics measures are periodically recorded.

AWS serves as a cold storage for old data. Data older than 7 days is converted to a parquet, deleted from Clickhouse, and uploaded to an S3 bucket.

  • Machine learning models are trained with recent market data on set intervals. Their performance are recorded, and the models are then uploaded to an AWS ML repository.
  • The ML models power several strategies which simulate real time trading. This process is powered via execution, risk, and portfolio monitoring engines.
